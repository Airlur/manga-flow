## 现有流程（当前代码实际流程）
1) 图片检测：`ImageDetector` 过滤图片并监听懒加载。
2) ROI：`TextDetector` 生成候选区域（可关闭/调试）。
3) OCR：
   - Cloud：Google Vision `TEXT_DETECTION`，取 `textAnnotations` 并按垂直距离合并。
   - Local：Tesseract.js，按 line 级 bbox 产出文本块。
4) 文本过滤：`TextFilter` 过滤水印/装饰/拟声词（拟声词硬过滤）、短词阈值放宽 + 组内保护。
5) 翻译：**按组批量翻译**（组内文本合并），随后 **拆回行级块** 用于渲染。
6) 图像擦除：`ImageProcessor` **按组 bbox**擦除，背景采样 + 像素替换 + 膨胀。
7) 渲染：`Renderer` **按组渲染**，组级字号统一、行内排版自适应、复杂背景遮罩增强。
8) 替换原图 + 缓存（阶段三已启用）。

> 说明：红框来自 OCR bbox，橙框为 ROI，绿框为 mask 参考框（调试可开关）。

## v0.2 新增：OCR 模型与方案评估（摘要）
### 现状结论
- 纯本地重模型（如 comic-text-detector / manga-ocr）对浏览器扩展体积与性能压力大。
- 云端 OCR（Google Vision）准确率高，但需结合裁剪识别与聚类，否则仍会漏字。

### 备选模型/方案方向
1) **Koharu**：桌面端 Rust/Tauri 应用，内部使用 `comic-text-detector`、`manga-ocr`、`AnimeMangaInpainting`，模型自动下载。优点是流程完整；缺点是模型体积与运行依赖较重，直接迁移到扩展成本高。  
2) **MeikiOCR**：偏日文视频游戏场景，水平文本专用；不适合作为韩文漫画主力模型。  
3) **PaddleOCR (PP-OCRv5/ONNX)**：模型较大（检测模型体积大），浏览器端可运行但需要较大带宽与内存；移动端压力大。  
4) **Browser OCR 封装库**（基于 PaddleOCR+ONNXRuntime）：可行但需评估模型体积与性能。  

结论：短期继续 **云端 OCR + 裁剪识别**，中期再评估轻量检测模型替换云端全图识别。

## 现有问题（重点）
0) **气泡语义判定缺失**
   - 当前仅用“背景复杂度”判定擦除/遮罩，非白气泡易误判为复杂背景。
   - 结果：气泡对白不擦除，改为遮罩/描边，观感不一致。

1) **遮罩宽度不稳定**
   - 遮罩宽度随译文长度动态扩展，出现“过长/过短”两端问题。
   - 结果：部分气泡漏盖原文，部分气泡遮罩溢出。

2) **ID/昵称语义翻译**
   - 短文本被当作普通对白翻译（如 ID/昵称被直译）。
   - 结果：语义翻译错误（应保留/音译）。

3) **拟声词/装饰文本分流不足**
   - 仅靠文本规则筛选，仍可能误识为对白。
   - 结果：拟声词被翻译或被错误擦除。

4) **特殊气泡/花体字识别不足**
   - 彩色/描边/变形文字在 Google Vision 下漏识或拆碎。
   - 结果：对应气泡漏翻/残字。

```text
  → 生成遮罩 + 膨胀
  → Inpaint 修复背景
  → 渲染译文（适配字号/描边/对比）
  → 替换原图 + 缓存
```

## 关键改进点与实现思路

### 1) 文本块聚类（气泡/段落合并）
**目标**：将同一气泡/旁白的多行文本合并成一个“组”，统一擦除 + 翻译。

**建议算法（轻量）**：
- 以行块为节点，按以下条件聚类：
  - y 轴重叠或垂直距离 < 行高 * 0.6
  - x 区域有重叠或中心距 < 块宽 * 0.6
  - 组内 bbox 合并成一个大 bbox
- 输出：
  - `groups[]`：每组包含文本列表 + union bbox
  - 翻译按组执行，渲染按组（再自动换行）

### 2) Mask 生成与膨胀
**目标**：保证擦除覆盖完整原文，而不是仅覆盖行 bbox。

**建议**：
- 以组 bbox 生成遮罩，`padding = max(行高*0.25, 4px)`
- 对 mask 做 1~2 次膨胀 (dilation)，覆盖笔画边缘
- 如仍残留，可增加二次膨胀或“软边缘羽化”

### 3) Inpaint 策略
**目标**：减少“译文叠字原文”。

**建议优先级**：
1. **inpaint.js / Telea**（优先）
2. 背景采样填充（fallback）
3. 透明遮罩 + 强描边文字（兜底）

**注意**：
- 单行 bbox 不适合 inpaint，优先使用“组 bbox”。
- 复杂背景可在少量区域使用强描边文字覆盖。

### 4) OCR 精度改进（不引入重模型）
**Cloud**：
- 尝试 `DOCUMENT_TEXT_DETECTION` 或 `fullTextAnnotation` 结构化解析（可选）。
- 若裁剪识别，则先聚类或用轻量检测得到 ROI，再调用 OCR。

**Local**：
- 继续保留 Tesseract，增加预处理（灰度/对比度/阈值）。
- 可选“轻量文本区域检测”：
  - 灰度 → 自适应阈值 → 闭运算 → 连通域 → 合并。

### 5) 过滤增强
- 加入站点水印关键词黑名单（如站点域名/常见广告词）。
- 按 bbox 位置过滤：底部固定区域/边角区域权重下降。

## 关键影响翻译效果的点（优先关注）
1) **bbox 合并质量** → 直接影响擦除完整度与翻译语义。
2) **mask 膨胀策略** → 决定是否“叠字”。
3) **背景复杂度判断** → 影响是否启用 inpaint 或 fallback。
4) **翻译粒度（按行 vs 按气泡）** → 影响上下文质量。

| 模块 | 现有流程 | 新流程建议 |
|------|----------|------------|
| OCR | 全图识别 + 行级 bbox | ROI 裁剪 + 行聚类/气泡合并 |
| 过滤 | 规则过滤 | 规则 + 位置 + 关键词 |

## 风险与注意事项
- ROI 裁剪可能导致 API 调用次数上升，需要控制阈值。
- inpaint 在复杂背景仍可能失败，需要 fallback 策略。
- 过度膨胀 mask 会擦掉气泡边框，需控制范围。

## v0.3 执行计划（结合最新测试反馈）

### 现象复盘（当前测试）
1) ROI 变成“整页大框”，导致裁剪范围过大、水印混入。
2) 缓存命中时 OCR 红框不再绘制，阶段 B 仅显示 ROI 框。
3) OCR 识别文本日志缺失，无法核对识别准确性。
4) 清除缓存按钮无效（只清除了错误的 storeName）。
5) 组级擦除仍有残字、个别气泡/水印被合并翻译。

### 优先级 P0（必须先解决）
1) **ROI 策略调整**：ROI 改为“气泡/文本块级”或暂时关闭 ROI，避免整页大框。
2) **OCR 文本块日志输出**：无论是否缓存命中，输出每块识别文本与 bbox。
3) **缓存命中仍绘制 OCR 框**：保证阶段 B 同时看到 ROI(橙) + OCR(红)。
4) **修复清除缓存逻辑**：清除 `ocr-cache` 与 `translation-cache` 两个 store。

### 优先级 P1（重要优化）
1) **水印/广告关键词过滤**：站点黑名单 + 关键词过滤。
2) **文本块拆分/分组修正**：根据行距/高度/位置拆分合并，减少“气泡+水印”混块。
3) **mask 扩边与组级合并**：提升擦除覆盖率，减少残字。

### 优先级 P2（体验优化，后做）
1) **自动翻译新加载图片**（可选开关）：默认关闭，避免测试时浪费 token。
2) **翻译批量策略微调**：根据可见图数量做节流。
3) **可视化诊断**：ROI/OCR/分组层级高亮开关。

### 体验建议（懒加载场景）
- 默认不自动翻译新图，保留“手动继续翻译”。
- 提供可选设置：自动翻译新图片（仅正式模式开启）。
- `batchSize` 主要控制并发，不解决懒加载本质问题。

### 过滤策略建议（记录）
1) **水印/广告**：直接过滤（黑名单关键词：站点域名/网址/广告词）。
2) **拟声词**：默认不翻译；提供可选开关（避免画面变丑）。
3) **非气泡文字**：短文本/被遮挡/场景标识优先过滤，必要时白名单启用。

### 优先级 P1（重要优化）
1) **特殊气泡 OCR 增强**：ROI 内预处理（灰度/对比度/放大/反色二次识别）。

3) **水印+正文混块拆分修正**：减少“气泡+水印”合并导致误删。

### 优先级 P2（体验优化）
1) **自动翻译新加载图片**（可选开关，默认关闭）。
2) **翻译节流策略**（按可见图数量）。
3) **per-block 交互编辑**（拖拽/缩放/透明度）。

## v0.5 执行计划（语义优先渲染策略）

### 目标
解决“气泡误判 → 不擦除”“遮罩宽度不稳”“ID/昵称误译”“拟声词误翻译”等问题。

### 核心策略
1) **语义分类优先（非 ROI、无模型）**  
   - **气泡对白**：优先擦除 + 干净渲染  
   - **ID/昵称标签**：保留原文/音译，弱化/不擦除  
   - **拟声词/装饰字**：不翻译、不擦除  
2) **遮罩宽度与气泡解耦**  
   - 气泡类：遮罩跟随气泡/文本区域，不再跟随译文长度  
   - 非气泡：遮罩跟随译文宽度 + 上限阈值  
3) **组内字号统一**  
   - 组内字号由“最长行宽度”约束统一，避免同气泡行间字号不一致  

### 优先级 P0（必须先做）
1) **气泡优先擦除规则**：在复杂背景判定之前，先做“气泡类”判断。  
2) **遮罩宽度稳定化**：气泡类遮罩跟随气泡框；非气泡按译文宽度 + 上限。  
3) **ID/昵称识别与保留策略**：短文本 + 位置 + 重复性判定为标签。  

### 优先级 P1（重要优化）
1) **拟声词/装饰字视觉规则**：加入位置、字体大小、描边强度等特征。  
2) **复杂背景对白兜底策略**：半透明遮罩 + 强描边，减少原文残留。  
3) **提示词辅助**：让 LLM 遇到 ID/昵称时优先保留或音译。  

### 优先级 P2（可延后）
1) **自动翻译新加载图片**（可选开关，默认关闭）。  
2) **翻译节流策略**（按可见图数量）。  
3) **per-block 交互编辑**（拖拽/缩放/透明度）。  
4) **文本/遮罩样式设定**（用户可选）。  
5) **轻量气泡检测（非模型）**：基于颜色集中度 + 边缘形状近似。

## v0.7 新增：本轮讨论结论与后续路线（2026-03-23）

### 一、当前问题重新归因
结合当前代码，现阶段体验差的根因不止是“Google OCR 不好”，而是：
1) **Google OCR 结果解析过粗**
   - 当前主流程仅使用 `TEXT_DETECTION + textAnnotations`
   - 代码侧又做了一次行合并，容易把 bbox 合并错
2) **擦除高度依赖 OCR bbox**
   - OCR 框一旦偏移，擦除就会漏盖、过盖、断行
3) **过滤、分类、渲染耦合仍较重**
   - 对话、拟声词、ID/昵称缺少稳定分流
4) **UI 与调试能力不够**
   - 用户缺少“局部修正”“失败提示”“阶段耗时”“原文/译文切换”等兜底体验

### 二、PaddleOCR 方向结论
结论：**值得做，且优先级高于继续死磕当前 Google OCR 实现。**

但模型选择建议调整：

#### 方案 A：推荐的韩漫主力轻量方案（优先）
- 检测模型：`PP-OCRv5_mobile_det`
- 识别模型：`korean_PP-OCRv5_mobile_rec`
- 文本行方向模型：`PP-LCNet_x0_25_textline_ori`

适合：
- 以韩语漫画为主
- 本地部署优先
- 后续可迁移到自建 API

#### 方案 B：更高精度方案（次优先）
- 检测模型：`PP-OCRv5_server_det`
- 识别模型：`korean_PP-OCRv5_mobile_rec`
- 文本行方向模型：`PP-LCNet_x1_0_textline_ori`

适合：
- 后续自建服务器
- 更看重检测效果
- 接受更高内存与推理开销

#### 方案 C：你当前提出的 PP-OCRv4 组合
- 检测模型：`ch_PP-OCRv4_det`
- 识别模型：`ch_PP-OCRv4_rec`
- 方向分类器：`ch_ppocr_mobile_v2.0_cls`

判断：
- **可用，但不建议作为韩漫主方案**
- 主要问题在于 `ch_PP-OCRv4_rec` 更偏中文通用，不是韩语优先模型
- 如果只是“先验证本地 OCR 服务链路”，可以短期试；如果是正式路线，不建议继续押它

### 三、关于“能不能一个模型通吃”
结论：**现实里不建议。**

原因：
1) 检测、识别、方向判断是三个不同任务
2) 漫画文字存在倾斜、竖排、拟声词、花字、描边字
3) 如果强行只留一个识别模型，方向错误与漏检问题会更明显

所以推荐：
- 最小闭环：`det + rec`
- 稳定产品方案：`det + rec + orientation`

### 四、关于本地部署与云端部署

#### 本地部署结论
你的设备：
- Intel Core i7-10870H
- NVIDIA GeForce GTX 1650
- 16GB 内存

结论：**能跑本地 PaddleOCR 服务，适合开发测试。**

建议：
- 第一阶段先只部署 OCR
- 第二阶段再评估 LaMa
- 模型目录允许放在 D 盘，不必绑在 E 盘项目目录

#### 云端部署结论
如果后续改为云端：

1) **仅 OCR 服务**
   - 2 vCPU / 4~8GB RAM 可起步
   - 若只跑轻量 PaddleOCR，CPU 机也能跑，但并发有限

2) **OCR + LaMa**
   - 更建议 GPU VPS
   - 起步建议 16GB 内存
   - 若仅自己用可低配试水，若多用户则需更高显存和更稳定网络
